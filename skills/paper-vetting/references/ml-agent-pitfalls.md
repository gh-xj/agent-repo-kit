# ML / LLM / Agent Paper Pitfalls

The dominant failure modes for ML/LLM/Agent papers in 2024–2026. Run these checks as part of Lens C whenever the paper is in this domain. Most are unique to ML and don't appear in general paper-evaluation rubrics.

## 1. Benchmark contamination (the #1 issue)

**The problem.** Public benchmarks (MMLU, HumanEval, GSM8K, BIG-Bench, ARC, etc.) leak into pretraining or post-training data. The model then "wins" the benchmark by memorizing it.

**How to detect from the paper:**

- Does the paper acknowledge contamination risk and report a _contamination check_?
- Does the model release date precede or follow the benchmark's public release? (Pre-existing benchmarks in the training cut-off → high risk.)
- Are the gains concentrated on benchmarks with leaked test sets, vs. held-out / dynamic / private evaluations?
- Does the paper rely on a single static benchmark for the headline claim, or does it include dynamic / time-cut / private-set evaluations?

**Red flags:**

- "+15% over baseline on MMLU" with no contamination analysis.
- All benchmarks predate the model's training cut-off and no held-out variant is reported.
- Benchmark was generated by an LLM (synthetic) — quality issues compound contamination.

**Reference:** _Benchmarking LLMs Under Data Contamination: A Survey from Static to Dynamic Evaluation_, ACL 2025 — confirms field is moving from static to dynamic benchmarks for this reason.

## 2. SOTA inflation via cherry-picked baselines

**The problem.** "We beat SOTA" where SOTA is an old model, a weak baseline, or a non-comparable setting (different params, different prompt budget, different inference cost).

**How to detect:**

- Are baselines from the same year, same compute scale, same prompt strategy?
- Does the paper compare against the _current_ leaderboard top, or a stale snapshot?
- Is inference cost (tokens, calls, wall-clock) reported alongside accuracy?

**Red flag:** Headline accuracy gain with no compute/latency normalization. A method that costs 10× the tokens for 2% better accuracy is not actually better.

## 3. Cherry-picked tasks / sub-benchmarks

**The problem.** Reporting only the subset of tasks where the method wins; dropping the ones where it loses.

**How to detect:**

- Is the _full_ benchmark suite reported, or only a subset?
- For agent benchmarks (SWE-bench, WebArena, GAIA, etc.) — are per-task scores or only the aggregate?
- Are there task families where the method underperforms? Are they discussed?

## 4. Single-seed / no-error-bar SOTA claims

**The problem.** ML training is high-variance; one good seed can produce a misleading number.

**How to detect:**

- Are results averaged over multiple seeds with std/CIs reported?
- For LLM judging, is the eval averaged over multiple prompt variants?
- Does the paper report best-of-N where N > 1 and disclose it?

## 5. LLM-as-judge contamination

**The problem.** Using a strong LLM (GPT-4, Claude, etc.) to score outputs of another LLM. The judge has its own biases (verbosity bias, self-bias, format bias) and is itself contaminated.

**How to detect:**

- Is human-in-the-loop validation reported on a sample?
- Does the paper acknowledge the judge's biases and run robustness checks (different judges, different prompts)?

## 6. Synthetic / LLM-generated benchmark data

**The problem.** Recent surveys show LLM-generated benchmarks produce unrealistic distributions and inflated headline numbers. Hand-written + human-reviewed benchmarks are the gold standard.

**How to detect:**

- How was the benchmark constructed? Hand-written / scraped / LLM-generated / hybrid?
- Was there independent human review?

## 7. No code / no weights / no configs

**The problem.** Method paper claims results that no one can reproduce.

**How to detect:**

- Code repo: present? last commit? stars? open issues? independent forks running it?
- Weights / checkpoints: released?
- Hyperparameters and prompts: in the paper or supplementary?
- For Agent papers: harness/scaffolding code released, or only model wrapper described?

**Red flag:** Headline result + "code coming soon" + no release date + paper >3 months old = probably never coming.

## 8. Reproducibility checklist mismatches

**The problem.** Authors check "yes" on the NeurIPS / ICML / ICLR reproducibility checklist while the paper body contradicts it.

**How to detect:**

- Find the checklist (usually at the end of the PDF).
- Cross-check each "yes" claim against the corresponding section. "Yes, code is included" should map to a working repo link.
- Look for missing / "n/a" entries on items that _should_ apply.

## 9. Prompt sensitivity hidden as method

**The problem.** Method = "use this clever prompt". Sensitivity to wording is enormous; the +5% gain is a prompt-tuning artifact, not a real algorithmic improvement.

**How to detect:**

- Is prompt sensitivity ablated? (Re-run with paraphrased prompts.)
- Does the method work on multiple base models, or only the one the prompt was tuned on?

## 10. Inference cost / latency / safety swept under the rug

**The problem.** Agent papers especially: a "better" agent that's 10× more expensive, 5× slower, or pays an unstudied safety cost.

**How to detect:**

- Cost-per-task reported? Token count? Wall-clock?
- Safety / robustness evaluation included, or out of scope?
- For Agent benchmarks: does the eval include adversarial scenarios?

**References:** _Evaluation and Benchmarking of LLM Agents: A Survey_ (arXiv 2507.21504); _Survey on Evaluation of LLM-based Agents_ (arXiv 2503.16416). Both flag cost-efficiency, safety, and robustness as systematically under-evaluated.

## 11. Survey-paper-specific risks

**The problem.** Surveys are easier to publish than original work. Common failures:

- Coverage breadth without analytical depth.
- Self-citation gravity (heavy reliance on the team's own prior work).
- Coining a new umbrella term for an existing pattern (then the term doesn't get adopted).
- Snapshot rapidly-decaying field — useful for 6 months, stale after 12.

**How to detect:**

- What % of citations are the team's own?
- Are the terms introduced novel, or do they already exist under different names?
- Date of v1 — for LLM/Agent surveys, anything >12 months old is partly historical.

## 12. Production-claim mismatch

**The problem.** Paper frames as "production system / harness / framework" but the team has no production deployment, no widely-used artifact, and no operational lessons.

**How to detect:**

- Does the team have a shipped, used product/library in this space?
- Are there post-deployment retrospectives, scaling studies, or incident reports?
- Or is the "production" claim entirely descriptive of literature?

## Quick Lens C ML checklist

Before believing any benchmark number in an ML/Agent paper:

- [ ] Was the benchmark public before the model's training cut-off?
- [ ] Is contamination addressed?
- [ ] Are baselines current and compute-comparable?
- [ ] Full benchmark suite reported, or sub-selection?
- [ ] Multi-seed with error bars?
- [ ] Code + weights + configs released?
- [ ] If LLM-judged: bias robustness reported?
- [ ] If prompt-based: prompt sensitivity reported?
- [ ] Inference cost / latency reported?
- [ ] Reproducibility checklist filled honestly and consistent with body?

If 6+ of these are "no" or unclear on a method paper, drop trust band by at least one letter regardless of the team or citation context.
